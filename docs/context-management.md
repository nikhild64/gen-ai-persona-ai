# Context Management — Chai Code Personas

_All thresholds live in `src/config/context-config.ts` per AD-9 — one file,
one canonical value each. ESLint bans re-declaration elsewhere._

## 1. Verbatim Tail + Rolling Summary + Drift Refresh

- **Verbatim Tail** (`VERBATIM_TAIL_LENGTH = 8`): the last 8 messages are
  always attached verbatim in AD-8 block 7 so short-range coherence never
  degrades.
- **Rolling Summary**: a compressed prose summary of everything before the
  tail, generated by the same provider that handles the persona's chat.
  Injected in AD-8 block 6.
- **Drift Refresh** (`DRIFT_REFRESH_FIRST_TURN = 15`,
  `DRIFT_REFRESH_CADENCE = 10`): a 60–100-token voice-rule reminder
  injected at turn 15 and every 10 turns thereafter, immediately before
  the current user message (block 8). Counters "lost-in-the-middle"
  attention loss.

## 2. Why not just longer verbatim tail?

Character.AI's public post-mortems flagged **~21 % recall at turn 40** as a
long-conversation failure mode. A pure verbatim tail eventually blows the
context window; a pure summary loses grounding. The **hybrid** — tail
for recency + summary for depth + drift-refresh for tone anchoring — is
the pattern this project follows.

## 3. Summary trigger (hybrid — AD-9)

- **Primary** trigger fires when
  `assistantMessageCount(thread) > VERBATIM_TAIL_LENGTH` AND
  `thread.turnsSinceLastSummary >= SUMMARY_REFRESH_CADENCE` (= 10).
- **Safety-net** trigger fires when
  `estimateTokens(projectedPrompt) > SUMMARY_TOKEN_BUDGET_PCT / 100 × context_window`
  (= 70 % of a 32 000-token window).
- **Ordering rule** with drift-refresh: on a turn where both would fire,
  summary runs first (persisted into the thread), and the next
  assembler call sees the fresh summary + injects the drift-refresh block.

## 4. Cost implications

- Rolling Summary is one extra LLM call per ~10 turns of chat.
- The summary is bounded at ~300 output tokens (`maxOutputTokens: 300` in
  the assembler's `composeSummary`).
- The Verbatim Tail is bounded at 8 messages — token cost per outbound
  prompt is predictable for budget headroom.

## 5. Token-budget math

- 40-turn conversation ≈ 40 × ~200 tokens = 8 000 tokens verbatim if we
  kept everything.
- Verbatim tail 8 × ~200 = 1 600 tokens.
- Rolling summary ~300 tokens.
- Identity + voice rules + refusal + few-shots + reminder + verification
  ≈ 2 500–3 000 tokens.
- Drift-refresh (when injected) ~100 tokens.
- **Total outbound prompt ≈ 5 500–5 800 tokens.**
- Well within the 30 %-headroom target (32 000 × 0.70 = 22 400 threshold)
  per PRD §4.4 NFR.

## 6. Client-side max-turn cap (AD-1 / FR-21)

- `MAX_TURNS_PER_THREAD = 40` — the app refuses to send a 41st message
  and renders the persona's `capRefusalTemplate` from `PERSONA_REGISTRY`.
- **Defeatable** by clearing IndexedDB — accepted trade-off of the Pure-FE
  topology (AD-1). The cap's role is user-cost-per-BYO-Key protection,
  not abuse prevention (abuse is naturally gated by the user needing
  their own paid provider quota).

## 7. Measurement

Long-conversation drift is measured by `bun run eval:drift`
(`evals/drift-curve.ts`) — 40-turn conversation per persona, sampled at
turns 5 / 15 / 25 / 35, scored against the 5-dimension rubric.

**SM-2 pass criterion:** turn-35 composite ≥ turn-5 composite − 0.45
(≈ 15 percentage points on a 3-point scale). Fail is a red flag for the
DRIFT_REFRESH_CADENCE + SUMMARY_REFRESH_CADENCE tuning; both live in
`context-config.ts` for easy eval-time adjustment.

Results captured in `evals/drift-curve.md`.
