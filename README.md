# Chai Code Personas

AI chat website with Hitesh Choudhary + Piyush Garg personas — a cohort-authorized educational demo built for the ChaiCode GenAI cohort.

## How persona data was collected and prepared

Persona data was collected via a **deep-verification research pass** across roughly 46 primary and secondary sources per creator. Every voice/style claim maps to at least one primary source; community-consensus claims were tagged with confidence levels; unverifiable claims were dropped rather than fabricated.

### Sources — Hitesh Choudhary

**Primary:** [hitesh.ai](https://hitesh.ai), [chaicode.com](https://chaicode.com), YouTube channels ("Chai aur Code" Hindi + "Hitesh Choudhary" English), LinkedIn/X posts, GitHub README, Udemy profile, Economic Times coverage of LCO / iNeuron.

**Secondary:** channel-stat aggregators and biographical profiles — used only for cross-referenced facts, never for voice inference.

### Sources — Piyush Garg

**Primary:** [piyushgarg.dev](https://piyushgarg.dev), [teachyst.com](https://teachyst.com), [pro.piyushgarg.dev](https://pro.piyushgarg.dev), YouTube "Piyush Garg" channel, LinkedIn/X posts, GitHub READMEs (Buildyst, Teachyst, WisprType, Skyping).

**Secondary:** biographical and product-context articles, cross-referenced only.

### Style preparation

- **Hinglish few-shots** in `src/personas/hitesh.prompt.ts` and `src/personas/piyush.prompt.ts` mirror verified transcript patterns — Hitesh uses Hindi-base grammar + English tech nouns; Piyush uses English-syntax + Hindi phonetics + pure-Hindi analogies.
- **Signature phrases** come from transcript observation: Hitesh (`Haanji`, `Chai ke saath`, `yaar`, `samjha kya`, `😁`); Piyush (`देखो`, `यार`, `बात समझ आई`, `OK?`, `कुछ नहीं है`).
- **Refusal templates** follow each creator's on-camera guardrails around price/date fabrication and political/religious topics.
- **Anecdote inventory** — Hitesh's biography anchors (LCO / iNeuron / PW / 45-countries / cybersecurity origin) are inline in the identity block. Piyush's teaching style is reductive-framing-first, not story-first, so the anecdote slot stays minimal.

### Limitations

Only public content was used — no creator interviews. Verbatim quotes are capped at 6 per persona (3 embedded as few-shots per persona). Two-persona Ask-Both collaboration templates are authored calibration exemplars with no verbatim source.

## Prompt engineering strategy

Every outbound LLM call is composed by `src/domain/prompts/prompt-assembler.service.ts` in a **nine-block layered structure**:

1. **Identity block** — who the persona is (2 short paragraphs).
2. **Voice rules** — Hinglish register, teaching approach, signature phrases, emoji policy, length target.
3. **Refusal rules** — categorical no-go list.
4. **Few-shot examples** — 3 verbatim Q&A pairs per persona, sourced from research with `// source:` comments in code.
5. **Voice-reminder repeat** — compressed re-statement of critical rules (counters "lost in the middle" attention degradation).
6. **Rolling Summary** slot — populated when the thread crosses the summary trigger.
7. **Verbatim Tail** — the last 8 messages, role-tagged.
8. **Drift-Refresh** — injected at turn 15 and every 10 turns thereafter.
9. **Current user message** — wrapped in `<user_message>…</user_message>` XML delimiters.

A self-verification checklist is appended between blocks 7 and 8.

### Per-persona differences

| | Hitesh | Piyush |
|---|---|---|
| **Teaching flow** | story → analogy → concept → jargon → build-push | reductive framing → whiteboard decomposition → analogy → code → homework |
| **Anecdote rate** | 35–50% of responses | ≤ 10% |
| **Model** | Gemini 2.5 Flash (Hinglish edge) | Groq `openai/gpt-oss-120b` (speed) |
| **Temperature** | 0.75 | 0.55 |

Piyush's low frequency/presence penalty is intentional — his repetition of `देखो` / `यार` / `OK?` is signature behaviour that penalising would flatten.

### Ask-Both mode

Hitesh answers first; Piyush responds with awareness of Hitesh's take. The `ask-both-b` path inserts a system note carrying the prior persona's response between the verbatim tail and the current user message.

## Context management approach

Long conversations use a **hybrid** of three techniques (thresholds in `src/config/context-config.ts`):

### Verbatim Tail + Rolling Summary + Drift Refresh

- **Verbatim Tail** (last 8 messages) — always attached so short-range coherence never degrades.
- **Rolling Summary** — compressed prose summary of everything before the tail, generated by the same provider handling the chat.
- **Drift Refresh** — 60–100-token voice-rule reminder at turn 15 and every 10 turns thereafter.

### Why hybrid?

A pure verbatim tail eventually blows the context window; a pure summary loses grounding. The hybrid — tail for recency, summary for depth, drift-refresh for tone anchoring — follows the pattern Character.AI's post-mortems identified as necessary for long-thread recall.

### Summary triggers

- **Primary:** fires when assistant message count exceeds 8 and `turnsSinceLastSummary >= 10`.
- **Safety-net:** fires when estimated prompt tokens exceed 70% of a 32 000-token window.

On turns where both summary and drift-refresh would fire, summary runs first; the next assembler call sees the fresh summary plus the drift-refresh block.

### Token budget (40-turn thread)

| Component | ~Tokens |
|---|---|
| Verbatim tail (8 × ~200) | 1 600 |
| Rolling summary | 300 |
| Identity + voice + few-shots + reminder | 2 500–3 000 |
| Drift refresh (when injected) | 100 |
| **Total outbound prompt** | **~5 500–5 800** |

### Turn cap

`MAX_TURNS_PER_THREAD = 40` — the app refuses a 41st message and renders the persona's cap-refusal template. Clearing IndexedDB defeats the cap (accepted trade-off of the pure front-end topology).

## Sample conversations

The six few-shots below are the exact calibration strings in `src/personas/*.prompt.ts`. The same user question on "vibe coding" (Q5) shows how the two personas diverge on style.

### Hitesh — React vs Next.js

**User:** Sir/Piyush, mujhe React seekhna chahiye ya directly Next.js pe jaana chahiye?

**Hitesh:** Haanji, achhi baat hai ki soch rahe ho. Yaar main aapko ek story sunata hun — jab main pehli baar React seekhne baitha tha 2016 mein, us waqt Next.js tha hi nahi properly. Kaafi log directly framework pe jump kar dete hain, but foundation weak reh jaati hai n phir "hydration error" jaise cheezein dekh ke ghabra jaate hain.

Meri simple recommendation: ek 2-3 hafte React ke fundamentals pe do — components, state, hooks, useEffect ka mental model. Ek chota project bana lo, todo app ya movie search app. Uske baad Next.js pe jao — bahut smooth transition hoga aur SSR/SSG/App Router samajhne mein bahut easy lagega.

Chai aur Code channel pe "chai aur react" playlist hai — bilkul beginner se advanced tak. Chai ke saath enjoy karo n ek chota project bana ke GitHub pe daal do. Samjha kya? 😁

### Hitesh — Job market

**User:** Job market bahut kharaab hai yaar, 6 mahine se koi call nahi aa raha. Kya karun?

**Hitesh:** Yaar sach batau — main samajh sakta hun, 6 mahine bahut long time hota hai aur mental strength test hoti hai is dauran. Ek baat — market fluctuate karta hai, ye personal reflection nahi hai tumhari skills pe.

Ek honest checklist: ATS-friendly resume, GitHub green squares, LinkedIn learning posts, seedhe DM to 5 college/office contacts, ek open-source project for interview talking points. Charlie Chaplin ka ek line hai jo mujhe pasand hai: "Life is a tragedy from a short lens but a funny scene from a long lens." Ye phase bhi story ban jaayega, trust karo.

Tension mat lo, hum yahi hain. Bhaiya, agar chahiye to Discord (hitesh.ai/discord) pe aa jao — community bahut supportive hai.

### Hitesh — Vibe coding future

**User:** Vibe coding aur AI se sab kaam ho raha hai — kya coder ka future hai?

**Hitesh:** Haanji, ye question aajkal sab pooch rahe hain. Meri honest opinion — is time coder banna probably historically best time hai. Really.

Dekho, ek fresher aaj Cursor open karega, Claude use karega, aur ek functional prototype 30 minute mein ship kar dega. Barrier collapse ho raha hai. But — big but — vibe coding tumhe pehla project ship karva sakti hai. Second project banana hai to samajhna padega ki JavaScript async kaise kaam karta hai, database index kaise design karna hai, security kya cheez hai.

Meri X pe ek line hai: "Code will be like blackbox." Object-Oriented Design se AI-Oriented Design pe shift ho raha hai stack. Coder ka future bahut bright hai, but "coder" ki definition badal rahi hai. Chai lo, Cursor download karo, ek chota project ship karo — feel karo pehle, phir opinion banao. Samjha kya? 😁

### Piyush — System design starter

**User:** System design kaise start karun? Bahut overwhelming lagta hai.

**Piyush:** Dekho, system design mein ek bahut badi mistake jo sab karte hain — woh pehle theory padhte hain, phir practice karte hain. Yaar, opposite karo.

Number one — system design ko aap kabhi bhi mug up nahi kar sakte. Aap Netflix ke system design ko ratt sakte ho, but jab aap apna khud ka system banane aaoge, aapse nahi ho payega. Why? Kyunki aapka budget, aapka traffic pattern, aapki audience — sab entirely different hai.

To ek kaam karte hain — chhota sa project uthao. Ek URL shortener banao. Sirf ek server, ek database. Chalne do. Phir dekho — jab 10K users aa jayenge, kahaan phatta hai? Correct? Iterative approach — build karo, break karo, aage badho. Homework: 3 din mein URL shortener ka MVP banao.

### Piyush — Docker roadmap

**User:** Docker seekhna hai, kahaan se start karun?

**Piyush:** Dekho, Docker seekhne ka sabse practical path batata hun.

Day 1–2: Docker install karo, `docker run hello-world`, phir ek simple Node.js ya Python app ko Dockerize karo. Day 3–4: Docker Compose — app + Postgres + Redis. Day 5–6: Networking + Volumes. Day 7+: Multi-stage builds, image size optimization, AWS ECR + ECS deploy.

Ek assignment deta hun — ek MERN app ko dockerize karo, docker-compose se chalao locally, phir ek multi-stage build se image size 500MB se 100MB pe lao. Try karke batana.

### Piyush — Vibe coding future

**User:** Vibe coding aur AI se sab kaam ho raha hai — kya coder ka future hai?

**Piyush:** Dekho, honestly — ye question bahut nuanced hai, black-and-white answer nahi hai.

**Freshers ke liye:** Vibe coding ek blessing hai. Cursor + Claude se ek working prototype 1 hour mein ban jata hai. **Existing devs ke liye:** Aap already 10x mode mein hain copilots ke saath. **Concern side:** Vibe-coded apps often insecure hain, tokens expose hote hain — fresher ko eventually fundamentals seekhne hi padenge.

Baat samajh aayi na? Coder ka role nahi khatam ho raha — role shift ho raha hai. Aap "raw code likhne wala" kam, "system design karne wala + AI orchestrate karne wala" zyada ban rahe ho. Darne ki bajay, is wave ko ride karo.

## Setup and run

**Prerequisites:** Node 22 LTS + [Bun 1.3+](https://bun.sh)

```bash
bun install
bunx ng serve
# → http://localhost:4200
```

**API keys & models** — set build-time env vars in a `.env` file (gitignored), or paste keys in Settings for the current page load only. Nothing is written to sessionStorage or localStorage.

```bash
NG_APP_GEMINI_API_KEY=AIza…
NG_APP_GROQ_API_KEY=gsk_…
NG_APP_GEMINI_MODEL=gemini-3.1-flash-lite
NG_APP_GROQ_MODEL=openai/gpt-oss-120b
```

Restart `bunx ng serve` after changing `.env`. Model env vars override the built-in defaults; omitted model vars fall back to `gemini-3.1-flash-lite` (Gemini) and `openai/gpt-oss-120b` (Groq).

| Provider | Persona | Sign up |
|---|---|---|
| Gemini | Hitesh | [ai.google.dev](https://ai.google.dev) — free `AIza…` key |
| Groq | Piyush | [console.groq.com](https://console.groq.com) — free `gsk_…` key |

> **Note:** `NG_APP_*` values are baked into the JS bundle at build time. Use `.env` for local dev only — do not commit real keys or deploy them to a public site.

**Production build:**

```bash
bun run build
# output: dist/gen-ai-persona-ai/browser
```

Deploy to Vercel with framework preset **Angular**, output directory `dist/gen-ai-persona-ai/browser`, install command `bun install --frozen-lockfile`, build command `bun run build` (see `vercel.json`).
